# Skill: Goal Clarification

Use this skill when the user's vision, idea, goal, ambition, or desired change is vague, too broad, unbounded, or missing success signals.

## Purpose

Turn a fuzzy wish into a goal contract the user can actually navigate toward.

## When To Use

- The user says "I want to become X", "I want to build Y", "I want to change Z", or "I have an idea" without a concrete outcome.
- The target is abstract and does not yet have visible success signals.
- The user has several possible directions and needs to choose.
- The user has not selected a guidance style.

## Inputs

- User's initial vision or wish.
- Known background and constraints.
- Available time, deadline, and preferred intensity if known.
- Any prior user profile or goal state.

## Process

1. Restate the user's wish in plain language.
2. Classify the goal category and mention the classification only if useful.
3. If the target is vague, offer 3 to 5 concrete destination shapes.
4. Ask one question at a time until the destination is concrete enough to assess.
5. Help the user choose a guidance style in user-facing language:
   - Relaxed companion
   - Direct mentor
   - Question-led coach
   - Action coach
   - Challenge mode
6. Convert the result into a goal contract.
7. Break the clarified goal into an Objective, 2 to 4 Success Signals, and observable outputs.

## Destination-Shaping Questions

Use the smallest useful set:

- What would be visibly different if this worked?
- What would make you say this was worth it?
- What outcome would feel wrong even if it looked successful?
- Which matters more right now: learning, shipping, deciding, changing a habit, or becoming a different kind of person?
- What constraint is non-negotiable?

## Output Format

```markdown
## Goal Contract Draft

- Goal category:
- Original vision:
- Clarified goal:
- Objective:
- Success Signals:
- Observable outputs:
- Time horizon:
- Weekly time budget:
- Constraints:
- Preferred guidance style:
- First review date:
- Open uncertainty:
```

## Quality Bar

A good goal contract answers:

- What exactly is the user trying to make true?
- What counts as success?
- Which Success Signals prove movement?
- What time and energy can they realistically invest?
- What type of output or behavior will prove progress?
- How should the agent guide them?

If these are not clear, keep clarifying before planning.
